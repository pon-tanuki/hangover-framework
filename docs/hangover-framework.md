# HANGOVERフレームワーク

**Human Audit Normalization for Generated Output, Verification, Enforcement, and Review**

> バイブコーディングの翌朝、あなたを救う5段階の回復プロセス。

作成日: 2026-03-07

---

## なぜHANGOVERか

**バイブコーディング**は最高の気分だ。「このボタンをいい感じにして」と言えば、AIが秒でコードを書いてくれる。デザイントークン？アクセシビリティ？知らない。とにかく動く。とにかく速い。

翌朝、あなたは画面を見る。

- `padding: 12px`（トークンは`--space-150`のはずだった）
- `color: #2462EA`（`#2563EB`からなぜか1ビットずれている）
- `font-family: Inter`（ブランドフォントはGaramondだった）
- コントラスト比: 2.8:1（WCAGは4.5:1以上）
- コンポーネント: 全て新規生成（既存の`<Button>`は無視）

**これがAIコーディングの二日酔いだ。**

HANGOVERフレームワークは、この二日酔いと体系的に戦うための5フェーズのプロセスである。楽しいバイブコーディングをやめろとは言わない。ただ、翌朝どう回復するかを知っておけ、という話だ。

---


## Phase 0: 飲む前に水を飲んでおく

### ツール: Design System Spec Compiler

**解決するギャップ: コンテキストの断片化 + 意図の構造化**

二日酔いを軽減する最善策は、飲む前に水を飲んでおくことだ。HANGOVERフレームワークにおける「水」とは、LLMが参照できる構造化されたデザインシステム仕様である。

Figma・Storybook・tokens.cssがバラバラに存在していても、LLMはそれを統合できない。Spec Compilerはこれらを**単一の機械可読仕様**にコンパイルする。

```
入力（バラバラな情報源）:
  ├── Figma Variables（色・タイポグラフィ・スペーシング）
  ├── Storybook Component Manifest（コンポーネントAPI）
  ├── tokens.css（デザイントークン実装）
  ├── 使用ガイドライン（Markdown）
  └── NG例（アンチパターン集）

出力（統合仕様）:
  └── design-system.context.md
      ├── トークン一覧（閉じた値セット）
      ├── コンポーネントカタログ（Props・バリアント・使用例）
      ├── 構成ルール（どのコンポーネントをどう組み合わせるか）
      ├── 禁止パターン（生のCSS値・非登録コンポーネント）
      └── ページテンプレート（レイアウトの骨格）
```

この仕様が`CLAUDE.md`から参照され、LLMの全セッションに注入される。

**効果の根拠:**
- Claude Skillsの実証: 400トークンの制約指示で**2-3倍の知覚品質向上**（Claude Blog）
- 4層アーキテクチャの実証: Atlaskitで418件の違反を**ゼロ**まで削減

**二日酔い軽減率: 高**（予防が最も安い）

## Phase 1: 夜通し飲む

### LLMコード生成（介入なし）

これがバイブコーディングだ。`design-system.context.md`が注入された状態でLLMに好きなだけコードを書かせる。

Phase 0があるだけで、LLMは：
- 存在するトークンを「知っている」
- 登録済みコンポーネントを「知っている」
- 禁止パターンを「知っている」

とはいえ、LLMはそれでも間違える。睡眠中に脳が勝手に動くように、LLMも確率的にトークンを捏造する。だから翌朝がある。

---

## Phase 2: 目が覚める

### ツール: Post-Generation Token Validator

**解決するギャップ: 生成後の即時検証欠如**

目覚めた瞬間の状態確認。二日酔いがひどいかどうか、最初に確認すること。

LLMがファイルを書き込んだ直後（Claude CodeのPostToolUseフック）に発動し、デザイントークン違反を**準リアルタイムで検知**する。

```
LLM → ファイル保存 → [PostToolUse Hook] → Token Validator
                                               │
                                     ┌─────────┴─────────────┐
                                     │  違反検出               │
                                     │                         │
                                     │  padding: 12px          │
                                     │  → var(--space-200)?    │  ← 提案
                                     │    (16px / 8px どちら?) │
                                     │                         │
                                     │  color: #2462EA         │
                                     │  → var(--color-primary) │  ← 提案
                                     └─────────────────────────┘
                                               │
                                     提案として表示（自動置換しない）
```

**なぜ自動置換しないか:**
`12px`が`--space-150`(8px)なのか`--space-200`(16px)なのか、文脈によって異なる。二日酔いの判断で薬を大量に飲むのは危険だ。まず状況を確認する。

**既存ツールとの違い:**

| ツール | 検知タイミング | フィードバックまで |
|--------|-------------|----------------|
| 監査スクリプト | CI実行時 | 分〜時間 |
| ESLint | 保存時 | 秒（ただしトークンマッピングなし） |
| **Token Validator** | 生成直後 | 秒（トークンマッピングあり） |

**二日酔い対処速度: 最速**

---

## Phase 3: 被害確認

### ツール: Design Quality Score (DQS)

**解決するギャップ: 出力検証の断片化**

起き上がって鏡を見る。全体的にどれだけひどいか確認する。

PRを出した時点でCI/CDが7つの品質次元を一括測定し、**単一スコア**を算出する。

```
PR作成 → GitHub Actions
              │
              ├─ axe-core ─────────── アクセシビリティ: 72/100
              ├─ Lighthouse ────────── パフォーマンス:   80/100
              ├─ Token Validator ───── トークン準拠率:   95/100
              ├─ Chromatic ─────────── ビジュアル一貫性: 85/100
              ├─ ESLint ────────────── コード構造:       70/100
              ├─ コンポーネント分析 ── 再利用率:         90/100
              └─ レスポンシブチェック  レイアウト:        65/100
                    │
                    ▼
            ┌─────────────────────────┐
            │  DQS: 78/100            │  ← PRに自動投稿
            │  閾値: 70 → PASS        │
            │  前PR比: +3             │
            └─────────────────────────┘
```

**「十分か」の判断を属人的にしない。** 70点以上ならマージ許可。それだけだ。

**現在のツール状況:**

| 既存ツール | 問題 |
|-----------|------|
| axe単体 | アクセシビリティのみ。他の次元は別途確認 |
| Lighthouse単体 | パフォーマンスのみ |
| ESLint単体 | コード品質のみ |
| **DQS** | 全てを統合。一つのスコアで判断 |

**二日酔い深刻度の定量化: 初めて可能になる**

---

## Phase 4: 家族への説明

### プロセス: AI Design Review Workflow

**解決するギャップ: レビュー観点の標準化**

自動チェックは「客観的な事実」しか検知できない。コントラスト比2.8は数値として測れる。しかし「このボタンはブランドらしくない」は測れない。

DQSが70以上であれば、トークン・アクセシビリティ・パフォーマンスは確認済みだ。レビュアーはUX・ブランド・意図の合致だけに集中できる。

```
DQS: PASS → デザインレビュー（人間）
                  │
                  ├─ □ ビジュアルヒエラルキーは適切か
                  ├─ □ ブランドガイドラインに沿っているか
                  ├─ □ ダークモードで問題ないか
                  ├─ □ インタラクション状態（loading/error/empty）は全て実装されているか
                  ├─ □ レスポンシブで崩れていないか
                  └─ □ セマンティックHTMLは適切か
                              │
                       承認 or 差し戻し
```

**マインドセットの転換:**

> 開発者は「コードの作成者」から「法医学的監査者」に切り替える必要がある。
> AIが生成したコードはUncanny Valley of Codeだ。ほぼ正しいが、微妙に違う。

**二日酔いのうち自動で解決できない部分: 60-70%**（axeとLighthouseで残り30-40%をカバー）

---

## Phase 5: 同じ過ちを繰り返す

### ツール: Cross-Session Consistency Tracker

**解決するギャップ: 継続的品質監視**

また飲む。また二日酔いになる。でも今度は記録がある。

複数のAIセッションにわたるドリフトを時系列で追跡する。各PRのDQSをgitコミット単位で記録し、品質劣化のタイミングと原因を特定する。

```
Session:  S1 ──── S2 ──── S3 ──── S4 ──── S5
DQS:      87      85      82      79      74 ← アラート発火
                                 │
                           ┌─────┴──────────────────┐
                           │ S3でspacingドリフト発生  │
                           │ padding: 12px が3件      │
                           │ → S3のコミットにロールバック可能  │
                           └───────────────────────────┘
```

**なぜ単発の品質ゲートでは不十分か:**

各PRが閾値70をクリアしていても、S1→S5で13ポイントのドリフトが起きている。5セッション後に「なんか違う」、10セッション後に「3チームが作ったように見える」現象は、個別PRの品質ゲートだけでは防げない。

**再発防止率: 記録がなければゼロ。記録があれば可能性が生まれる。**

## フェーズまとめ

| フェーズ | 二日酔いのフェーズ | ツール | 検知タイミング | 修正コスト |
|---------|-----------------|-------|-------------|----------|
| **0** | 飲む前に水を飲む | Spec Compiler | 生成前（予防） | ゼロ |
| **1** | 夜通し飲む | LLM生成 | - | - |
| **2** | 目が覚める | Token Validator | 生成直後（秒） | 極小 |
| **3** | 被害確認 | DQS | PR作成時（分） | 小 |
| **4** | 家族への説明 | Design Review | レビュー時（時間） | 中 |
| **5** | 同じ過ちを繰り返す | Consistency Tracker | マージ後（日〜週） | 大 |

**原則: 左で捕まえるほど安い。** Phase 0で予防できればPhase 5の問題は発生しない。

---

## ツール間の依存関係

```
Phase 0: Spec Compiler（起点。他の全てが依存する）
    │
    ▼
Phase 2: Token Validator（context.mdからトークンマッピングを読み込む）
    │
    ▼
Phase 3: DQS（トークン準拠率はToken Validatorの結果を含む）
    │
    ├──→ Phase 4: Design Review（DQSスコアをPRに自動投稿）
    │
    ▼
Phase 5: Consistency Tracker（DQSをセッション単位で記録）
```

> **警告**: Phase 5（Consistency Tracker）はプロジェクト開始時から運用ルールを設計する必要がある。後付け導入は困難。二日酔いの記録は、最初から付けておかなければ意味がない。

## 実装の優先順位

| 優先度 | ツール/プロセス | 実装難易度 | 理由 |
|--------|--------------|----------|------|
| **最高** | Phase 0: Spec Compiler | 中 | 他の全ての基盤。これがなければ他が機能しない |
| **高** | Phase 2: Token Validator | 低〜中 | 即効性が高く、修正ラウンドトリップを大幅削減 |
| **高** | Phase 4: Design Review | 低 | チェックリストとPRテンプレートのみ。即日導入可能 |
| **中** | Phase 3: DQS | 中〜高 | 既存ツール（axe/Lighthouse）の統合が必要 |
| **中** | Phase 5: Consistency Tracker | 中 | 長期運用が前提。早期着手が望ましい |

## 根本的な限界（正直に言う）

HANGOVERフレームワークは、バイブコーディングをなくさない。なくせない。

このフレームワークが目指すのは、**LLMを「ジュニアアシスタント」として扱い、明確な制約と検証の中で活用する**ことだ。最も成功している対策は全て、LLMの出力を人間が設計した制約の中に閉じ込めることで品質を担保している。

LLMに完全な自律性を与えることは、現時点では選択肢にない。

> "AI-generated UI curse" は今日も続いている。HANGOVERは、その呪いと付き合いながら生き延びるための方法論だ。
